## 1. HTTP 请求结构

HTTP 请求通常由以下几部分组成：

```http
POST /v1/chat/completions HTTP/1.1
Host: api.example.com
Authorization: Bearer sk-xxx
Content-Type: application/json
Idempotency-Key: req-xxx

{
  "model": "gpt-x",
  "messages": [
    {"role": "user", "content": "hello"}
  ]
}
````

核心组成：

```text
Method   请求方法，例如 GET / POST / DELETE
Path     请求路径，例如 /v1/chat/completions
Headers  请求元信息，例如 Authorization / Content-Type
Body     请求体，例如 JSON 参数
```

---

## 2. HTTP Method 语义

| Method | 语义          | 是否幂等 |
| ------ | ----------- | ---- |
| GET    | 读取资源        | 是    |
| POST   | 创建资源 / 执行动作 | 通常不是 |
| PUT    | 整体替换资源      | 是    |
| PATCH  | 局部更新资源      | 不一定  |
| DELETE | 删除资源        | 通常是  |

幂等性含义：

```text
幂等不是没有副作用，而是重复执行多次，最终状态一致。
```

例子：

```http
DELETE /v1/files/file_123
```

执行一次或多次，最终都是文件不存在，所以通常认为是幂等。

```http
POST /v1/chat/completions
```

重复执行可能会重复推理、重复扣费，所以默认不是幂等。

对于 POST 重试，可以引入：

```http
Idempotency-Key: req-xxx
```

服务端用 `Idempotency-Key` 做去重，避免客户端超时重试导致重复生成或重复扣费。

# 3. curl 基础

`curl` 是命令行 HTTP 客户端，用来手动构造 HTTP 请求。

基本格式：

```bash
curl [options] URL
```

## 3.1 GET 请求

```bash
curl -i "https://httpbin.org/get"
```

含义：

```text
-i   显示响应状态行和响应 Header
URL  请求目标
```

---

## 3.2 POST JSON 请求

```bash
curl -X POST "https://httpbin.org/post" \
  -H "Content-Type: application/json" \
  -H "X-Request-ID: req-001" \
  -d '{"prompt":"a futuristic city","size":"1024x1024"}'
```

参数解释：

```text
-X POST   指定 HTTP Method
-H        添加 Header
-d        添加请求 Body
```

注意：

```text
-d 只是发送 body，不代表 body 一定是 JSON。
如果 body 是 JSON，需要配合：
Content-Type: application/json
```

---

## 3.3 常用 curl 参数

| 参数                  | 含义             |
| ------------------- | -------------- |
| `-X`                | 指定 HTTP method |
| `-H`                | 添加 HTTP header |
| `-d`                | 发送请求 body      |
| `-i`                | 显示响应 header    |
| `-v`                | 显示请求调试信息       |
| `-sS`               | 静默输出，但保留错误     |
| `-o`                | 保存响应到文件        |
| `-w`                | 请求结束后输出统计信息    |
| `-L`                | 跟随重定向          |
| `--connect-timeout` | 连接超时           |
| `--max-time`        | 整个请求最大耗时       |
| `--retry`           | 请求失败重试         |
| `-N`                | 关闭输出缓冲，常用于 SSE |
| `-F`                | multipart 文件上传 |
| `-G`                | 构造 GET query   |

---

## 3.4 curl `-w` 查看耗时

`-w` 是 `write-out`，用于请求结束后打印统计信息。

最小实用版：

```bash
curl -sS -o /dev/null \
  -w "status=%{http_code} ttfb=%{time_starttransfer}s total=%{time_total}s\n" \
  "https://httpbin.org/get"
```

更完整版本：

```bash
curl -sS -o /dev/null \
  -w "status=%{http_code} dns=%{time_namelookup}s connect=%{time_connect}s tls=%{time_appconnect}s ttfb=%{time_starttransfer}s total=%{time_total}s\n" \
  "https://httpbin.org/get"
```

字段含义：

| 字段                   | 含义             | 单位 |
| -------------------- | -------------- | -- |
| `http_code`          | HTTP 状态码       | 无  |
| `time_namelookup`    | DNS 解析完成时间     | 秒  |
| `time_connect`       | TCP 连接完成时间     | 秒  |
| `time_appconnect`    | TLS 握手完成时间     | 秒  |
| `time_starttransfer` | 收到第一个字节时间，TTFB | 秒  |
| `time_total`         | 整个请求完成时间       | 秒  |

注意：

```text
curl 的 time_* 字段是从请求开始累计到某阶段完成的时间点，不是每个阶段的独立耗时。
```

阶段耗时计算：

```text
DNS 阶段耗时       = time_namelookup
TCP 阶段耗时       = time_connect - time_namelookup
TLS 阶段耗时       = time_appconnect - time_connect
服务端处理/等待     = time_starttransfer - time_appconnect
响应传输耗时        = time_total - time_starttransfer
```

# 4. REST API 表达

REST API 的核心：

```text
用 URL 表达资源
用 HTTP Method 表达动作
用 HTTP Status Code 表达结果
```

## 4.1 URL 应该像资源，不像函数

推荐：

```http
GET /v1/models
GET /v1/files/file_123
POST /v1/files
DELETE /v1/files/file_123
```

不推荐：

```http
GET /v1/getModels
POST /v1/createFile
POST /v1/deleteFile
```

原因：

```text
动作已经由 HTTP Method 表达，不需要塞进 URL。
```

---

## 4.2 Collection 和 Resource

```text
collection：资源集合
resource：具体资源
```

例子：

```http
GET /v1/files
```

表示获取文件列表，`/files` 是 collection。

```http
GET /v1/files/file_123
```

表示获取某个文件，`file_123` 是具体 resource。

---

## 4.3 AI Infra 常见 API

```http
GET /v1/models
```

查询模型列表。

```http
GET /v1/models/gpt-x
```

查询某个模型。

```http
POST /v1/files
```

上传文件，创建文件资源。

```http
GET /v1/files/file_123
```

查询文件信息。

```http
DELETE /v1/files/file_123
```

删除文件。

```http
POST /v1/chat/completions
```

创建一次模型推理请求。

---

## 4.4 REST 和 RPC 的区别

REST 更像：

```text
对一个资源做操作
```

例如：

```http
DELETE /v1/files/file_123
```

RPC 更像：

```text
调用一个函数
```

例如：

```http
POST /v1/generateImage
POST /v1/createEmbedding
POST /v1/rerank
```

AI Infra 中：

```text
模型、文件、任务、fine-tuning job：更适合 REST 资源风格
chat completion、embedding、rerank、image generation：经常是动作型 endpoint
```

表达方式：

```text
资源型对象用 REST；
动作型推理接口可以使用 RPC-style endpoint；
但仍然通过 HTTP method、status code、headers、body 表达语义。
```

# 5. HTTP 状态码

HTTP 状态码表达：

```text
请求在 HTTP 层面是成功、客户端错误，还是服务端错误。
```

## 5.1 2xx 成功

| 状态码          | 含义        | 场景               |
| ------------ | --------- | ---------------- |
| 200 OK       | 请求成功      | 查询模型、非流式推理成功     |
| 201 Created  | 资源创建成功    | 上传文件、创建任务        |
| 202 Accepted | 请求已接收但未完成 | 异步图片生成、batch job |

示例：

```http
HTTP/1.1 201 Created
Location: /v1/files/file_123
```

```http
HTTP/1.1 202 Accepted
```

```json
{
  "job_id": "job_123",
  "status": "queued"
}
```

---

## 5.2 4xx 客户端错误

| 状态码                      | 含义         | 场景                       |
| ------------------------ | ---------- | ------------------------ |
| 400 Bad Request          | 请求格式或参数错误  | JSON 错误、参数缺失、参数越界        |
| 401 Unauthorized         | 未认证 / 认证失败 | 没带 API key、API key 无效    |
| 403 Forbidden            | 已认证但没权限    | 没有访问某个模型权限               |
| 404 Not Found            | 资源不存在      | 文件不存在、模型不存在              |
| 409 Conflict             | 资源状态冲突     | job 已 running，不能重复 start |
| 422 Unprocessable Entity | 语义校验失败     | JSON 合法但业务规则不合法          |
| 429 Too Many Requests    | 触发限流       | RPM / TPM / 并发超限         |

401 和 403 区别：

```text
401：你是谁，我还没确认。
403：我知道你是谁，但你不能访问这个资源。
```

---

## 5.3 5xx 服务端错误

| 状态码                       | 含义         | 场景                                 |
| ------------------------- | ---------- | ---------------------------------- |
| 500 Internal Server Error | 内部未知错误     | 未分类服务端异常                           |
| 502 Bad Gateway           | 网关收到上游无效响应 | Gateway 调 inference service 返回异常   |
| 503 Service Unavailable   | 服务暂时不可用    | 模型 worker 过载、GPU 资源不足              |
| 504 Gateway Timeout       | 网关等待上游超时   | API Gateway 等 inference service 超时 |

关键区别：

```text
500：服务端内部未知错误
503：服务暂时不可用 / 过载
504：网关等待上游超时
```

429 和 503 区别：

```text
429：客户端 / 租户 / API key 维度触发限流
503：服务端 / 系统容量维度暂时不可用
```

# 6. SSE 基础

SSE 全称：

```text
Server-Sent Events
```

SSE 本质：

```text
一个长时间不结束的 HTTP response，body 格式是 text/event-stream。
```

普通 HTTP JSON：

```text
client -> request
server -> response headers + full body
server -> response end
```

SSE：

```text
client -> request
server -> response headers
server -> body event 1
server -> body event 2
server -> body event 3
...
server -> response end
```

---

## 6.1 SSE 连接过程

SSE 和普通 HTTP 连接建立过程一样：

```text
TCP 三次握手
TLS 握手，如果是 HTTPS
客户端发送 HTTP request
服务端返回 HTTP response headers
服务端持续写 response body
```

SSE 没有额外的握手。

---

## 6.2 SSE 响应头

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

如果经过 Nginx，可以加：

```http
X-Accel-Buffering: no
```

用于关闭代理缓冲，避免 token 被代理层攒起来再发。

---

## 6.3 SSE 数据格式

最小格式：

```text
data: hello

data: world

data: [DONE]

```

每个事件用空行结束：

```text
data: xxx\n\n
```

`\n\n` 是 SSE event 的边界。

---

## 6.4 SSE 字段

| 字段      | 含义                |
| ------- | ----------------- |
| `data`  | 事件数据              |
| `event` | 事件类型，自定义          |
| `id`    | 事件 ID，断线重连时使用     |
| `retry` | 断线后客户端多久重连，单位 ms  |
| `:`     | 注释行，常用于 heartbeat |

示例：

```text
event: token
id: 101
data: {"delta":"hello"}

event: token
id: 102
data: {"delta":" world"}

event: done
data: {"finish_reason":"stop"}

```

---

## 6.5 event 字段

`event` 没有固定枚举，是服务端自定义的事件类型。

例如：

```text
event: token
event: usage
event: done
event: server_error
event: progress
```

客户端根据事件名监听。

如果不写 `event`，默认事件类型是：

```text
message
```

示例：

```text
data: hello

```

等价于默认 `message` 事件。

---

## 6.6 data 字段

`data` 是业务数据。

```text
data: hello

```

如果有多行 `data`：

```text
data: hello
data: world

```

客户端会把它合并为：

```text
hello
world
```

传 JSON 时，通常一行一个 JSON：

```text
data: {"delta":"hello"}

```

---

## 6.7 id 和 Last-Event-ID

服务端可以发送：

```text
id: 20
event: progress
data: {"percent":60}

```

如果连接断开，浏览器 `EventSource` 重连时会自动带：

```http
Last-Event-ID: 20
```

服务端可以根据 `Last-Event-ID` 判断客户端最后收到哪个事件。

适合：

```text
日志流
通知流
任务进度流
股票价格流
系统事件流
```

AI token streaming 通常不强依赖 `id` 续传，因为模型生成状态不一定容易恢复。

---

## 6.8 retry 字段

`retry` 用于告诉客户端：

```text
如果 SSE 连接断开，多久之后自动重连。
```

单位是毫秒。

```text
retry: 5000

```

表示连接断开后，客户端等待 5000ms 后重连。

注意：

```text
retry 是让客户端重连，不是让服务端重试，也不是让模型重试。
```

`retry` 主要对浏览器原生 `EventSource` 有用。

如果使用 `fetch + ReadableStream`，需要客户端自己解析 `retry` 并实现重连逻辑。

---

## 6.9 Heartbeat

以 `:` 开头的是注释行，不触发业务事件。

```text
: ping

```

常用于 heartbeat，防止中间代理、负载均衡器或 NAT 因连接 idle 而断开。

---

# 7. SSE 和 HTTP chunk / HTTP2 DATA frame

SSE 是应用层事件格式：

```text
data: {"delta":"hello"}\n\n
```

HTTP/1.1 传输 response body 时，常用 chunked transfer：

```text
19\r\n
data: {"delta":"hello"}\n\n
\r\n
```

HTTP/2 没有 chunked，而是用 DATA frame 承载 body 字节。

分层理解：

```text
SSE：定义 response body 里的字节如何解释成事件
HTTP chunk / HTTP2 DATA frame：定义 response body 的字节如何被传输
TCP：负责可靠字节传输
```

关键结论：

```text
HTTP chunk 和 SSE event 没有一一对应关系。
```

一个 SSE event 可能被拆成多个 HTTP chunk。

一个 HTTP chunk 也可能包含多个 SSE event。

SSE event 的边界由：

```text
\n\n
```

决定，不由 TCP 包、HTTP chunk 或 HTTP2 DATA frame 决定。

# 8. SSE 服务端实现要点

伪代码：

```python
async def stream_chat(request):
    validate(request)

    await send_headers({
        "Content-Type": "text/event-stream",
        "Cache-Control": "no-cache",
    })

    async for token in model.generate_stream(request):
        event = f"data: {json.dumps({'delta': token})}\n\n"
        await response.write(event)
        await response.drain()

    await response.write("data: [DONE]\n\n")
```

关键：

```text
write   写入事件
flush   及时推给客户端
drain   处理 backpressure
```

如果只 write 不 flush，数据可能被框架、代理、TLS 或客户端缓冲，导致用户看不到实时 token。

---

## 8.1 Backpressure

如果客户端读取很慢，服务端不能无限写入内存。

需要处理：

```text
等待 socket 可写
限制缓冲区大小
取消生成
断开连接
释放 GPU / KV cache / request state
```

---

## 8.2 客户端断开

用户关闭页面或网络中断时，服务端要检测断连，并取消模型推理任务。

否则会浪费：

```text
GPU decode slot
KV cache
worker 并发
计费状态
队列资源
```

---

## 8.3 SSE 高并发

SSE 是长连接，会长期占用资源：

```text
TCP connection
file descriptor
TLS state
socket buffer
HTTP request context
应用层 session
网关连接槽位
```

AI 场景还会占用：

```text
GPU 推理并发
KV cache 显存
decode slot
scheduler queue
token queue
```

如果是传统 blocking thread-per-request 模型：

```text
1 个 SSE 连接 ≈ 1 个 worker thread 长期占用
```

高并发下容易耗尽线程池。

生产中更适合：

```text
async I/O
event loop
goroutine
coroutine
非阻塞 socket
```

同时要做：

```text
用户级并发限制
组织级并发限制
max stream duration
idle timeout
heartbeat
断连清理
backpressure
队列超时
```

# 9. SSE 客户端用法

## 9.1 curl 调试 SSE

```bash
curl -N -i "https://example.com/events"
```

`-N` 表示 no-buffer，关闭 curl 输出缓冲。

AI 推理接口：

```bash
curl -N -i -X POST "https://api.example.com/v1/chat/completions" \
  -H "Authorization: Bearer sk-xxx" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{
    "model": "gpt-x",
    "stream": true,
    "messages": [
      {"role": "user", "content": "hello"}
    ]
  }'
```

---

## 9.2 浏览器 EventSource

原生 `EventSource` 主要用于 GET SSE：

```js
const es = new EventSource("/events");

es.onmessage = (event) => {
  console.log("message:", event.data);
};

es.addEventListener("token", (event) => {
  console.log("token:", JSON.parse(event.data));
});

es.onerror = (err) => {
  console.error("sse error", err);
};
```

服务端发：

```text
event: token
data: {"delta":"hello"}

```

客户端监听：

```js
es.addEventListener("token", handler);
```

---

## 9.3 fetch + ReadableStream

AI API 常用 POST + JSON body，因此经常不用原生 `EventSource`，而是用 `fetch + ReadableStream`。

原因：

```text
EventSource 主要是 GET
EventSource 不方便带 JSON body
EventSource 不方便自定义 Authorization header
AI Chat 请求通常需要 POST body
```

客户端需要自己：

```text
持续读取 response body
按 \n\n 切分 SSE event
解析 event / data / id / retry 字段
遇到 [DONE] 结束
```

# 10. 错误码设计

真实 API 通常有两层错误码：

```text
HTTP status code：表达通用 HTTP 语义
业务 error code：表达具体业务原因
```

示例：

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 10
X-Request-ID: req_abc
```

```json
{
  "error": {
    "code": 429002,
    "reason": "token_rate_limit_exceeded",
    "message": "Token rate limit exceeded.",
    "request_id": "req_abc"
  }
}
```

---

## 10.1 字段设计

推荐错误对象：

```json
{
  "error": {
    "code": 400001,
    "reason": "context_length_exceeded",
    "message": "The messages exceed the model context window.",
    "param": "messages",
    "request_id": "req_123"
  }
}
```

字段含义：

| 字段           | 含义                |
| ------------ | ----------------- |
| `code`       | 稳定业务错误码，可以是数字或字符串 |
| `reason`     | 稳定、可读、机器可识别的错误原因  |
| `message`    | 给开发者看的自然语言说明      |
| `param`      | 哪个参数出错            |
| `request_id` | 排障和链路追踪           |

---

## 10.2 code 可以是数字，也可以是字符串

数字 code：

```json
{
  "code": 400102,
  "message": "context length exceeded"
}
```

优点：

```text
短
稳定
适合内部系统
```

缺点：

```text
可读性差
需要查错误码表
```

字符串 code：

```json
{
  "code": "context_length_exceeded",
  "message": "The input exceeds the model context window."
}
```

优点：

```text
可读
稳定
适合开放 API / SDK
```

更推荐的折中设计：

```json
{
  "code": 400102,
  "reason": "context_length_exceeded",
  "message": "The input exceeds the model context window."
}
```

程序判断应该依赖：

```text
HTTP status + code / reason
```

不应该依赖 `message`，因为 message 可能改文案、改语言、改格式。

---

## 10.3 常见错误映射

| 场景               | HTTP status | code / reason               |
| ---------------- | ----------: | --------------------------- |
| 没带 API key       |         401 | `missing_api_key`           |
| API key 无效       |         401 | `invalid_api_key`           |
| API key 有效但无模型权限 |         403 | `model_access_denied`       |
| 模型不存在            |         404 | `model_not_found`           |
| 参数越界             |   400 / 422 | `invalid_parameter`         |
| 上下文超长            |   400 / 422 | `context_length_exceeded`   |
| RPM 超限           |         429 | `rate_limit_exceeded`       |
| TPM 超限           |         429 | `token_rate_limit_exceeded` |
| worker 过载        |         503 | `model_overloaded`          |
| 上游超时             |         504 | `upstream_timeout`          |
| 内部未知错误           |         500 | `internal_error`            |

---

## 10.4 是否可重试

不建议重试：

```text
400 invalid_parameter
400 context_length_exceeded
401 invalid_api_key
403 model_access_denied
404 model_not_found
```

原因是请求本身有问题，直接重试通常无效。

可以重试，但要延迟：

```text
429 rate_limit_exceeded
429 token_rate_limit_exceeded
500 internal_error
502 bad_gateway
503 model_overloaded
504 upstream_timeout
```

重试策略：

```text
优先遵守 Retry-After
使用 exponential backoff
添加 jitter
POST 请求配合 Idempotency-Key
```

示例：

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 10
```

表示客户端至少 10 秒后再重试。

# 11. SSE 错误处理

SSE 错误分两类：

```text
建流前错误
建流后错误
```

---

## 11.1 建流前错误

如果还没有发送 SSE response header，可以直接返回 HTTP status + JSON error。

例如没带 API key：

```http
HTTP/1.1 401 Unauthorized
Content-Type: application/json
```

```json
{
  "error": {
    "code": 401001,
    "reason": "missing_api_key",
    "message": "Missing API key.",
    "request_id": "req_123"
  }
}
```

限流：

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 10
Content-Type: application/json
```

```json
{
  "error": {
    "code": 429002,
    "reason": "token_rate_limit_exceeded",
    "message": "Token rate limit exceeded.",
    "request_id": "req_123"
  }
}
```

---

## 11.2 建流后错误

如果已经发送：

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
```

就不能再修改 HTTP status 为 500 / 503。

只能在 SSE body 中发送错误事件：

```text
event: server_error
data: {"code":503001,"reason":"model_overloaded","message":"Model worker overloaded.","request_id":"req_123"}

```

然后关闭连接。

关键结论：

```text
建流前错误：HTTP status code + JSON error
建流后错误：SSE event-level error + close stream
```

# 12. Chat Completion API 设计

## 12.1 Method 和 Path

```http
POST /v1/chat/completions
```

原因：

```text
Chat completion 是一次模型推理动作，不是读取已有资源。
请求体包含 model、messages、temperature、stream 等结构化参数。
messages 可能很长且敏感，不适合放在 GET query 中。
```

---

## 12.2 Request Headers

```http
Authorization: Bearer sk-xxx
Content-Type: application/json
Accept: application/json
Idempotency-Key: req-xxx
X-Request-ID: req-abc
```

字段含义：

| Header            | 含义                |
| ----------------- | ----------------- |
| `Authorization`   | 鉴权                |
| `Content-Type`    | 请求体格式             |
| `Accept`          | 客户端希望的响应格式        |
| `Idempotency-Key` | 防止重试导致重复扣费 / 重复任务 |
| `X-Request-ID`    | 链路追踪              |

流式请求：

```http
Accept: text/event-stream
```

---

## 12.3 Request Body

```json
{
  "model": "gpt-x",
  "messages": [
    {"role": "user", "content": "hello"}
  ],
  "temperature": 0.7,
  "stream": true
}
```

字段含义：

| 字段            | 含义     |
| ------------- | ------ |
| `model`       | 选择模型   |
| `messages`    | 对话上下文  |
| `temperature` | 采样参数   |
| `stream`      | 是否流式返回 |

---

## 12.4 非流式返回

请求：

```json
{
  "stream": false
}
```

响应：

```http
HTTP/1.1 200 OK
Content-Type: application/json
X-Request-ID: req_123
```

```json
{
  "id": "chatcmpl_123",
  "model": "gpt-x",
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "Hello!"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 10,
    "completion_tokens": 3,
    "total_tokens": 13
  }
}
```

---

## 12.5 流式 SSE 返回

请求：

```json
{
  "stream": true
}
```

响应：

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
X-Request-ID: req_123
```

body：

```text
data: {"choices":[{"delta":{"role":"assistant"}}]}

data: {"choices":[{"delta":{"content":"Hello"}}]}

data: {"choices":[{"delta":{"content":"!"}}]}

data: {"choices":[{"finish_reason":"stop"}]}

data: [DONE]

```

客户端：

```text
持续读取 response body
按 \n\n 切分 event
解析 data 字段
遇到 [DONE] 结束
```
