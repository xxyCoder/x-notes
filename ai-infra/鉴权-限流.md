## 1. 基本概念

### 1.1 Authentication：鉴权

鉴权解决的问题是：

> 这个请求是谁发来的？

在 API 场景中，常见方式是 API key：

```http
Authorization: Bearer sk_xxx
````

服务端通过 API key 识别：

```text
user_id
org_id
project_id
service_account
environment
```

鉴权失败通常表示：

```text
API key 不存在
API key 已过期
API key 被禁用
API key 被撤销
API key 校验失败
```

常见返回：

```http
401 Unauthorized
```

---

### 1.2 Authorization：授权

授权解决的问题是：

> 这个身份能不能访问这个资源？

例如：

```text
是否允许调用 /v1/chat/completions
是否允许调用 qwen-72b
是否允许使用 embedding
是否允许创建 API key
```

授权依赖鉴权后的身份信息：

```text
user_id
org_id
project_id
scopes
allowed_models
roles
policies
```

授权失败通常返回：

```http
403 Forbidden
```

---

### 1.3 Quota：总量限制

Quota 控制长期总用量。

常见形式：

```text
每月最多 1000 万 tokens
每天最多 10 万次请求
每月最多消费 100 美元
每个项目每天最多生成 500 张图
```

Quota 关注的是：

```text
某个周期内总共用了多少
```

常见维度：

```text
requests
tokens
cost
GPU seconds
images
audio minutes
```

---

### 1.4 Rate Limit：速率限制

Rate Limit 控制短时间内的请求速度。

常见形式：

```text
60 RPM
1000 RPS
90k TPM
```

含义：

```text
RPM = requests per minute
RPS = requests per second
TPM = tokens per minute
```

Rate Limit 关注的是：

```text
当前这一小段时间内是不是请求过快
```

---

### 1.5 Concurrency：并发限制

Concurrency 控制当前正在运行的请求数量。

例如：

```text
某用户最多同时运行 5 个 qwen-72b 请求
```

它关注的是：

```text
当前有多少请求正在推理中
```

不是单位时间请求数。

通常使用：

```text
semaphore
```

---

## 2. API Key 设计

### 2.1 API key 不是简单字符串

API key 背后通常对应一条数据库记录：

```text
key_id
user_id
org_id
project_id
status
scopes
allowed_models
rate_limit_policy
quota_policy
expires_at
created_at
last_used_at
```

示例：

```text
key_id          = key_123
user_id         = user_001
org_id          = org_a
project_id      = proj_gateway
status          = active
scopes          = chat:write, embedding:read
allowed_models  = qwen-7b, gpt-4o-mini
rate_limit      = 60 RPM
quota           = 10M tokens / month
expires_at      = 2026-12-31
```

---

### 2.2 API key 不应该明文存储

服务端不应该保存完整明文 key。

推荐方式：

```text
创建 key 时只展示一次完整 key
服务端保存 secret hash
请求进来后对 secret 做 hash 校验
```

原因：

```text
即使数据库泄露，攻击者也不能直接拿 API key 调用服务
```

---

### 2.3 推荐结构

API key 可以设计成：

```text
sk_live_keyid_secret
```

其中：

```text
sk_live     = prefix，用于识别环境或类型
keyid       = 用于快速定位数据库记录
secret      = 真正的随机密钥
```

服务端流程：

```text
1. 从 API key 中解析 key_id
2. 根据 key_id 查数据库记录
3. 对 secret 做 hash
4. 和数据库中的 secret_hash 比较
5. 校验 status、expires_at、revoked 等状态
```

---

### 2.4 API key 应该按服务隔离

不同服务不应该共用同一个 API key。

例如：

```text
service-a：线上 chat API
service-b：后台 embedding 任务
```

应该分别使用：

```text
service-a → 独立 key，只允许 chat
service-b → 独立 key，只允许 embedding
```

这样可以做到：

```text
调用来源可追踪
计费可区分
权限最小化
泄露影响范围更小
key rotation 更安全
```

---

## 3. Token 计量

### 3.1 LLM token 类型

LLM 场景中不只有 prompt token 和 completion token。

常见 token 类型：

```text
input_tokens
cached_input_tokens
non_cached_input_tokens
output_tokens
reasoning_tokens
audio_tokens
image_tokens
embedding_tokens
tool_tokens
```

---

### 3.2 Cached Token

cached token 是命中缓存的输入 token。

例如：

```text
input_tokens = 50,500
cached_input_tokens = 50,000
non_cached_input_tokens = 500
```

cached token 可以单独计量，因为它的计算成本通常低于普通 input token。

计费或 quota 可以有不同策略：

```text
cached token 也计入 quota
cached token 不计入 quota
cached token 低价计入 cost quota
```

---

### 3.3 Reasoning Token

reasoning token 是推理模型内部思考消耗的 token。

用户可见输出可能很短：

```text
output_tokens = 200
```

但模型内部可能消耗：

```text
reasoning_tokens = 2000
```

因此结算不能只看用户可见输出。

---

### 3.4 Billable Usage

生产系统通常会计算 billable usage：

```text
billable_tokens =
  non_cached_input_tokens
+ output_tokens
+ reasoning_tokens
```

也可以直接转换为：

```text
actual_cost_usd
```

对于多模型、多价格、多 token 类型的系统，按 cost 做 quota 更准确。

---

## 4. Quota 设计

### 4.1 Quota 的作用

Quota 用于控制长期总用量。

例如：

```text
org 每月最多 1 亿 tokens
project 每天最多 100 万 tokens
user 每月最多消费 100 美元
```

Quota 可以按不同维度设置：

```text
user
org
project
api_key
model
endpoint
```

---

### 4.2 请求前预留

LLM 请求执行前，系统通常只能知道最大可能消耗：

```text
input_tokens
max_output_tokens
max_reasoning_tokens
```

因此可以先计算：

```text
estimated_billable_tokens =
  non_cached_input_tokens
+ max_output_tokens
+ max_reasoning_tokens
```

请求进入推理前先预留额度。

---

### 4.3 请求后结算

推理结束后，系统拿到真实 usage：

```text
actual_input_tokens
actual_cached_input_tokens
actual_output_tokens
actual_reasoning_tokens
actual_cost
```

再计算真实消耗：

```text
actual_billable_tokens =
  non_cached_input_tokens
+ output_tokens
+ reasoning_tokens
```

然后进行对账：

```text
reserved > actual  → 返还差额
reserved < actual  → 补扣或记录超额
reserved = actual  → 正常结算
```

这种模式称为：

```text
reservation / reconcile
```

---

## 5. Rate Limit 设计

### 5.1 常见限流维度

AI Infra 中常见限流维度：

```text
api_key
user_id
org_id
project_id
model
endpoint
IP
region
tenant tier
```

示例：

```text
api_key:sk_xxx       60 RPM
project:proj_a       300 RPM
org:org_a            3000 RPM
model:qwen-72b       1000 global RPM
ip:1.2.3.4           120 RPM
```

---

### 5.2 多维限流

一个请求通常需要同时通过多个限流规则：

```text
api_key bucket
user bucket
org bucket
model bucket
IP bucket
```

只要任意一个维度超限，请求就应该被拒绝。

多维限流需要保证：

```text
check all → deduct all
```

即：

```text
先检查所有 bucket 是否足够
全部足够才一起扣减
任意不足则全部不扣
```

---

### 5.3 RPM 和 TPM

RPM 用于限制请求数：

```text
cost = 1
```

TPM 用于限制 token 吞吐：

```text
cost = estimated_billable_tokens
```

例如：

```text
cost =
  non_cached_input_tokens
+ max_output_tokens
+ max_reasoning_tokens
```

---

## 6. Fixed Window

### 6.1 定义

Fixed Window 按固定时间窗口计数。

例如：

```text
每分钟最多 60 次
```

窗口是自然切分的：

```text
12:00:00 ~ 12:00:59
12:01:00 ~ 12:01:59
```

---

### 6.2 特点

优点：

```text
实现简单
性能高
适合粗粒度 quota
```

缺点：

```text
存在边界突刺
```

例如：

```text
12:00:59 发 60 次
12:01:00 发 60 次
```

固定窗口会认为两个窗口都没有超限，但实际 1 秒内来了 120 次。

---

### 6.3 适用场景

适合：

```text
daily quota
monthly quota
后台统计限制
低成本接口粗略限流
```

不适合：

```text
高成本在线推理入口
强实时保护后端的 API
```

---

## 7. Sliding Window

### 7.1 定义

Sliding Window 统计最近一段时间内的请求数。

例如：

```text
任意连续 60 秒内最多 60 次
```

它关注的是：

```text
过去 60 秒内已经发生了多少请求
```

---

### 7.2 额度释放方式

Sliding Window 中，请求会在窗口内占用名额。

只有当旧请求滑出窗口后，它占用的名额才会释放。

因此：

```text
释放速率不固定
```

例如：

```text
t=0 一次性放行 60 个请求
```

那么在接近 t=60 之前，这些请求仍然占用最近 60 秒窗口内的名额。

---

### 7.3 特点

优点：

```text
比 Fixed Window 更准确
可以避免窗口边界突刺
适合严格限制最近时间窗口内的行为
```

缺点：

```text
实现成本更高
高 QPS 场景下存储和计算压力更大
```

---

### 7.4 适用场景

适合：

```text
登录失败次数
短信验证码发送
创建 API key
敏感管理操作
风控行为限制
```

---

## 8. Token Bucket

### 8.1 定义

Token Bucket 维护一个令牌桶。

核心参数：

```text
capacity       桶容量
tokens         当前令牌数
refill_rate    令牌补充速率
last_refill_at 上次补充时间
```

请求需要消耗令牌：

```text
tokens >= cost → 放行，并扣减 tokens
tokens < cost  → 拒绝或排队
```

---

### 8.2 令牌补充

令牌根据时间均匀补充：

```text
new_tokens = min(capacity, tokens + elapsed * refill_rate)
```

例如：

```text
capacity = 10
refill_rate = 1 token/s
```

表示：

```text
每秒补 1 个 token
最多攒 10 个 token
```

---

### 8.3 特点

Token Bucket 允许短时突发。

例如：

```text
60 RPM
refill_rate = 1 token/s
capacity = 10
```

含义：

```text
长期平均每秒 1 个请求
短时间最多可以突发 10 个请求
```

---

### 8.4 适用场景

适合：

```text
API Gateway
在线 LLM API
API key RPM
org TPM
模型级吞吐保护
```

LLM TPM 中，cost 通常不是 1，而是：

```text
cost = estimated_billable_tokens
```

---

## 9. Leaky Bucket

### 9.1 定义

Leaky Bucket 更像队列。

请求进入桶，系统以固定速度处理。

```text
请求进入队列
worker 按固定速率消费
队列满了则拒绝
```

---

### 9.2 特点

Leaky Bucket 关注的是：

```text
平滑输出
```

它可以把突发输入变成稳定处理速率。

例如：

```text
用户瞬间提交 1000 条 embedding 任务
队列最多放 500 条
GPU worker 每秒处理 100 条
```

这里：

```text
桶容量 = 500
漏出速率 = 100 条/s
超出容量 = 拒绝或丢弃
```

---

### 9.3 适用场景

适合：

```text
embedding batch
离线推理
批量数据处理
异步任务
```

不适合：

```text
在线 chat completion 长时间排队
```

---

## 10. 限流算法对比

| 算法             | 核心含义       | 额度恢复方式        | 是否允许突发     | 适合场景                 |
| -------------- | ---------- | ------------- | ---------- | -------------------- |
| Fixed Window   | 统计自然窗口内请求数 | 窗口切换时重置       | 有边界突刺      | daily/monthly quota  |
| Sliding Window | 统计最近窗口内请求数 | 旧请求过期后释放      | 较严格        | 登录、验证码、敏感操作          |
| Token Bucket   | 看当前令牌余额    | 固定速率补充        | 允许         | 在线 API、LLM Gateway   |
| Leaky Bucket   | 队列固定速率出队   | worker 固定速率消费 | 输入可突发，输出平滑 | embedding batch、离线任务 |

记法：

```text
Fixed Window：数自然窗口
Sliding Window：数最近窗口
Token Bucket：看令牌余额
Leaky Bucket：排队固定出水
```

## 11. 原子性

限流判断和扣减必须原子。

非原子流程可能出现超卖：

```text
tokens = 1

请求 A 读到 tokens = 1
请求 B 读到 tokens = 1

A 放行
B 放行
```

最终实际放行了 2 个请求，但系统只有 1 个 token。

生产中常见方式：

```text
Redis Lua
Redis Function
数据库行锁
专门 RateLimit Service
```

核心要求：

```text
读状态
计算 refill
判断是否足够
扣减
写回
```

必须作为一个不可插入的整体执行。

## 12. 并发限制

### 12.1 并发限制模型

并发限制使用 semaphore。

维护状态：

```text
inflight_requests
```

请求开始：

```text
inflight_requests += 1
```

请求结束：

```text
inflight_requests -= 1
```

如果：

```text
inflight_requests >= limit
```

则拒绝或排队。

---

### 12.2 并发限制的资源含义

LLM 请求运行期间会占用：

```text
GPU compute
KV cache
scheduler slot
streaming connection
worker slot
```

因此即使 RPM 没超，也可能因为并发过高导致系统不可用。

---

### 12.3 Lease / TTL

并发 slot 需要防止泄露。

异常情况包括：

```text
请求超时
客户端断开
网关崩溃
推理服务失败
streaming 中断
```

生产中通常需要：

```text
request_id
lease
TTL
心跳
过期回收
```

---

## 13. 限流失败响应

### 13.1 状态码

常见状态码：

```text
401 Unauthorized       API key 缺失或无效
403 Forbidden          有身份但无权限
429 Too Many Requests  超过限流
503 Service Unavailable 后端容量不足或依赖故障
```

限流失败通常返回：

```http
429 Too Many Requests
```

---

### 13.2 响应头

常见限流响应头：

```http
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1710000000
Retry-After: 10
```

用途：

```text
客户端退避重试
客户端排队
客户端降级模型
用户提示稍后再试
```

---

## 14. Redis 故障策略

### 14.1 Fail-open

限流系统故障时放行。

```text
Redis 不可用 → 请求继续通过
```

适合：

```text
内部可信服务
低成本接口
强可用性优先场景
```

风险：

```text
可能打爆后端
可能产生成本失控
可能放大攻击流量
```

---

### 14.2 Fail-closed

限流系统故障时拒绝。

```text
Redis 不可用 → 返回 429 / 503
```

适合：

```text
外部公开 API
免费用户
高成本模型
强预算控制场景
```

---

### 14.3 分层降级

AI Infra 中可以按风险分层：

```text
qwen-72b + 外部免费用户 + 高成本 → fail-closed
内部服务 + 低成本模型 → 本地保守限流后 fail-open
企业付费用户 → 根据 SLA 和成本策略决定
```

---

## 15. LLM Gateway 完整流程

### 15.1 请求进入

```http
POST /v1/chat/completions
Authorization: Bearer sk_xxx

{
  "model": "qwen-72b",
  "messages": [...]
}
```

---

### 15.2 鉴权

流程：

```text
1. 解析 Authorization
2. 提取 key_id / prefix / secret
3. 根据 key_id 查数据库记录
4. 校验 secret hash
5. 检查 key status
6. 检查 expires_at / revoked
7. 得到 user_id / org_id / project_id / policy
```

鉴权失败：

```http
401 Unauthorized
```

---

### 15.3 授权

检查：

```text
是否允许访问 /v1/chat/completions
是否允许调用 qwen-72b
API key scope 是否包含 chat:write
project 是否允许使用该 endpoint
org 是否允许使用该模型
```

授权失败：

```http
403 Forbidden
```

---

### 15.4 Quota 预留

计算预估消耗：

```text
estimated_billable_tokens =
  non_cached_input_tokens
+ max_output_tokens
+ max_reasoning_tokens
```

检查：

```text
月度 token quota
budget quota
project quota
org quota
```

额度足够则预留。

额度不足则拒绝。

---

### 15.5 RPM / TPM 限流

API key RPM：

```text
维度：api_key
算法：Token Bucket / GCRA
cost = 1
```

Org TPM：

```text
维度：org_id
算法：Token Bucket
cost = estimated_billable_tokens
```

多维限流：

```text
api_key
user
org
project
model
IP
```

要求：

```text
check all → deduct all
```

---

### 15.6 并发限制

用户并发：

```text
维度：user_id
机制：semaphore
状态：inflight_requests
```

流程：

```text
1. 请求进入推理前尝试获取并发 slot
2. 获取成功则进入推理
3. 获取失败则拒绝或排队
4. 请求结束后释放 slot
```

---

### 15.7 推理执行

进入后端前记录：

```text
request_id
user_id
org_id
project_id
model
reserved_tokens
start_time
```

推理期间占用：

```text
GPU compute
KV cache
scheduler slot
streaming connection
concurrency slot
```

---

### 15.8 结束结算

推理完成后拿到真实 usage：

```text
input_tokens
cached_input_tokens
non_cached_input_tokens
output_tokens
reasoning_tokens
actual_cost
latency
```

计算实际消耗：

```text
actual_billable_tokens =
  non_cached_input_tokens
+ output_tokens
+ reasoning_tokens
```

对账：

```text
reserved_tokens > actual_billable_tokens → 返还差额
reserved_tokens < actual_billable_tokens → 补扣或记录超额
reserved_tokens = actual_billable_tokens → 正常结算
```

同时执行：

```text
释放并发 slot
写 usage log
写 billing log
写 audit log
更新 metrics
```

## 16. 在线请求与离线任务

### 16.1 在线 Chat API

特点：

```text
用户正在等待响应
延迟敏感
连接占用资源
```

适合：

```text
Token Bucket / GCRA
超限直接返回 429
带 Retry-After
```

不适合长时间排队。

---

### 16.2 离线 Embedding / Batch

特点：

```text
任务量大
可接受延迟
适合异步处理
```

适合：

```text
队列
Leaky Bucket
worker 按固定速率消费
```

---
